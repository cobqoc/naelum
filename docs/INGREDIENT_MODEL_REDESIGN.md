# 재료 모델 재설계 — 정체성 + 분류 + 관계 그래프 (정본)

> **이 문서의 위치**: `docs/INGREDIENT_MATCHING_REDESIGN.md`(2026-05-29)의 **"왜(§1)"·"정직성 원칙"·"회귀 체크리스트(§5)"는 유효**하나, 그 문서가 제안한 **`forms[]` jsonb 모델은 본 문서가 대체(supersede)**한다. 이유는 §7. 작성: 2026-05-31.

---

## 0. 한 줄 요약

재료의 **정체성·분류·관계를 데이터로 명시**하고, 매칭은 그 데이터를 **결정적으로 조회**만 한다. 추측 0.
모델 = **메타데이터 컬럼**(정체성·별칭·분류·알레르기) + **관계 엣지 테이블**(가공·대체). `forms[]`는 채택하지 않는다.
**모양이 명확한 인프라는 지금**(분류 계층 base_id → 양·단위 비교 → 대체 비율, §8), **데이터·그리고 모양이 미정인 둘(소비 자동 추적·대체 맥락)만 나중**(§8-B·§9-B).

## 1. 설계 원칙 (불변)

1. **추측하지 않는다.** 같으면 같다고, 다르면 다르다고 *데이터에* 있다. 문자열 substring·접두사 분리·정규화 추측으로 매칭하지 않는다.
2. **신뢰 > 매칭 풍부함.** 거짓 "만들 수 있어요"는 빈 결과보다 나쁘다. 불확실하면 missing.
3. **두 레이어를 분리한다.**
   - **Resolution** (쓰기 시점): 텍스트 이름 → 재료 id. 레시피 저장·냉장고 추가.
   - **Matching** (읽기 시점): 사용자 id 집합 vs 레시피 id 집합 → owned/preparable/substitute/missing.
   둘은 책임이 다르다. resolution이 id를 정확히 붙이고, matching은 id만 본다.
4. **"모양이 정해졌으면 지금, 모양이 미정이면 미룬다."** ← 핵심 판단 기준.
   - *모양 명확*(스키마·매칭·변환 수학·어드민)은 데이터가 없어도 **지금 만든다**. degrade-safe라 데이터 들어올 때 자연히 작동. 재료 66개일 때가 1,600개일 때보다 싸다.
   - *모양 미정*(써봐야 정해지는 UX·아직 분류축이 안 잡힌 구조)은 지금 만들면 **틀린 모양을 짓고 갈아엎는다(rework)** → 미룬다. ⚠️ "데이터가 없어서"는 미룰 이유가 *아니다*(degrade로 해결). 미루는 진짜 이유는 *모양 미정*뿐.
   - 데이터 입력 자체는 양에 비례하니 미뤄도 단가 불변 → 큐로 점진.
5. **additive·degrade-safe.** 새 컬럼이 null·정보 없으면 기존과 동일 동작. feature flag 없이 점진 점등 가능.

## 2. 재료 사이 관계 — 5종 분류 (모델의 출발점)

미래에 어떤 재료가 와도 아래 5종 안에 들어간다. 모델은 이 5종을 **각각 명확히** 표현해야 한다.

| # | 관계 | 정의 | 예 | 매칭 의미 |
|---|---|---|---|---|
| 1 | **동일 (별칭)** | 같은 물건, 다른 표기/언어 | 통마늘=garlic=蒜, "다진 마늘"="다진마늘" | 같은 id로 **resolution** |
| 2 | **분류 (부위·종류, is-a)** | A는 B의 한 종류 | 삼겹살→돼지고기, 청양고추→고추, 현미→쌀 | 변형 보유 → base 필요 = **owned** (단방향) |
| 3 | **가공 (form)** | A를 손질하면 B | 마늘→다진마늘, 쌀→밥 | from 보유 → to 필요 = **preparable** (단방향) |
| 4 | **대체** | A를 B로 바꿔 써도 됨 | 버터↔마가린, 액젓끼리 | **substitute** (양방향) |
| 5 | **알레르기 성분** | A는 알레르겐 X를 포함 | 땅콩버터 → [땅콩, 우유] | 안전 차단 |

추가로 매칭엔 *쓰지 않지만* 미래 가치가 있는 6번 — **구성 (made-from)**: 토마토소스는 토마토로 만듦. 정직성상 **자동 매칭 금지**(토마토 있다고 토마토소스 보유 아님). §9 미래 확장에서 다룬다.

### 핵심 통찰

- **1·2·5는 1:다 또는 1:1** 구조다 → **컬럼/배열**이 자연스럽다(별칭 배열, 단일 부모 base_id, 알레르겐 배열).
- **3·4는 다:다 + 승인 필요** → **엣지 테이블**(`ingredient_relations`)이 자연스럽다.
- **2(분류)와 3(가공)은 다르다.** 삼겹살은 돼지고기의 *한 종류*(is-a)지 *가공형*이 아니다. 다진마늘은 마늘의 *가공형*(preparable)이지 *종류*가 아니다. 섞으면 매칭 방향이 틀어진다.

## 3. 데이터 모델 — 하이브리드 (컬럼 + 엣지)

```
ingredients_master  (정체성 + 분류 + 메타데이터)
  id                    재료 고유 번호 (정체성)
  name                  표준 이름
  name_en / name_ko     표시·i18n
  aliases[]             동일(별칭) — 검색·resolution fallback        [관계 1]
  base_ingredient_id    분류 — "이 재료는 <base>의 한 종류" self-FK   [관계 2]  ★신규
  allergens[]           이 재료의 알레르겐 (변형은 base에서 상속)      [관계 5]
  category, emoji, …    기존 (도감·표시)
  is_processed          기존 (도감 '가공식품' 탭 필터 — 매칭과 무관)

ingredient_relations  (가공·대체 그래프)
  from_id, to_id
  kind ∈ { preparable_to(단방향), substitute(양방향) }              [관계 3·4]
  source, suggestion_count, approved_by, approved_at
```

**왜 컬럼과 엣지를 섞나 (full 엣지 통합 안 함):**
- 별칭(1)은 resolution이 *빠른 이름→id 맵*으로 매번 로드한다 → 배열 컬럼이 단순·빠름. 엣지면 매 resolution이 그래프 순회.
- 분류(2)는 *부모가 정확히 하나* → self-FK 컬럼이 제약을 자연 강제 + "돼지고기의 모든 변형" 쿼리가 `WHERE base_id=…`로 싸다.
- 가공·대체(3·4)는 다:다 + 어드민 승인 큐가 필요 → 엣지 테이블이 맞다 (이미 존재).

즉 **각 관계를 가장 잘 맞는 자료구조에** 둔다. 과도한 추상화(전부 엣지)도, 과도한 구겨넣기(`forms[]` jsonb)도 아니다.

## 4. Resolution 레이어 (텍스트 → id) — *이미 구현됨*

> 2026-05-31 구현 완료. `lib/ingredients/resolveIngredientId.ts` `resolveExactIngredientId(s)`.

쓰기 시점(레시피 저장·냉장고 추가·장보기→냉장고)에서 이름을 **결정적으로** id로 해석한다. 우선순위:

1. **정확 이름** 일치 (승인 마스터 `name`)
2. **별칭** 일치 (`aliases[]` — 큐레이션된 동의어)
3. **공백 무시** 정확 일치 ("다진 마늘"→"다진마늘"; "마늘"로 오연결 안 됨)

접두사 분리·단어 쪼개기 같은 fuzzy는 **하지 않는다**. 위에 안 걸리면 `ingredient_id = null` → **어드민 "번호 연결" 큐**로 흐른다.

냉장고·레시피·장보기 **세 경로 모두 동일 해석기**를 쓴다(비대칭 제거 완료). → 같은 입력은 어디서든 같은 id.

**미해결(null) = 데이터 일거리지 버그가 아니다.** 큐에서 사람이 ① 기존 id에 연결 ② 별칭으로 연결 ③ 새 마스터 생성 중 택해 처리.

## 5. Matching 레이어 (id → 종류) — 결정적 우선순위

레시피 재료 R(하나의 id)을 사용자 보유 집합 U(각 보유 재료는 자기 `base_id`를 앎)와 대조:

```
1. R ∈ U                         → owned          (정확 보유)
2. ∃ u∈U. u.base_id == R         → owned (변형)    [신규] 삼겹살 보유 → "돼지고기" 필요 충족
3. substitute 엣지 (u ↔ R)        → substitute     양방향 대체
4. preparable 엣지 (u → R)        → preparable     마늘 보유 → "다진마늘" 필요 = 가공 가능
5. 그 외                          → missing
```

`물·생수` 등 **기본재료(fundamental)**는 누구나 보유로 간주(코드 상수, 카운트 제외) — 기존 유지.

### 방향성 규칙 (정직성의 핵심)

- **변형 → base = 매칭 (2번).** 삼겹살은 돼지고기다. "돼지고기 필요" 레시피에 삼겹살 보유는 충족.
- **base → 변형 = missing.** 일반 "돼지고기"만 있을 때 "삼겹살 필요" 레시피는 *보장 안 됨* → missing. (옵션으로 "거의/대체 후보" 노출 가능하나 **기본은 missing** — 신뢰 우선.)
- **형제 변형끼리 = missing.** 목살 보유 → "삼겹살 필요" = 다른 부위라 missing. (정말 호환되면 둘 사이 `substitute` 엣지를 *명시적으로* 추가.)
- **가공 역방향 = missing.** 다진마늘 보유 → "통마늘 필요" = 되돌릴 수 없음 (기존 단방향 유지).
- **전이(transitivity) 없음 — 직접 엣지만 본다 (정직성).** 통마늘→편마늘, 편마늘→다진마늘 두 엣지가 있어도 통마늘→다진마늘은 *자동으로 성립 안 함*. 체인을 추적하면 추측이 섞인다(버터↔마가린·마가린↔쇼트닝 ≠ 버터↔쇼트닝). preparable·substitute 모두 동일. → **필요하면 통마늘→다진마늘을 *직접* 엣지로 추가**(8-B 데이터 규율). `forms[]`를 버리며 잃은 체인 세분화는 이렇게 *명시적 직접 엣지*로 메운다 — 단순·정직.

### 결과 인터페이스 (UI 영향 최소)

기존 `MatchKind = owned|preparable|substitute|missing` 유지. 변형 매칭은 **owned로 분류**하되 `via`에 보유 재료 id를 담아 칩에 "삼겹살로" 표시 가능. → `RecipeCard 배지·IngredientsTab·useCartFromRecipe` 등 호출처는 구조 변경 없음.

### 양(quantity) 레이어 — 위 종류 판정 *위에* 얹음 (§8 Phase 2)

위 1~5는 *종류*(있나/만들 수 있나)를 정한다. 그 **다음에** 양을 본다: owned/변형/대체로 판정된 재료에 한해, 레시피·냉장고 양이 *둘 다 있고 단위가 같거나 변환 가능*하면 `enough`/`short(부족분)`을 덧붙인다. 하나라도 없거나 변환 불가면 **양 판단 생략(=종류 결과 그대로)** — 거짓 정확성 회피. `MatchKind`는 불변, `shortBy?` 메타만 선택적 추가.

**어디서 적용되나 (혼동 방지):**
- **추천 목록**: 인분 선택 UI가 없으므로 *레시피 기본 인분* 기준. 양은 **표시·랭킹 보조**로만(부족분 적은 순 살짝 우대) 쓰고, "거의 가능/바로 가능"의 1차 분류는 *여전히 종류(missing 개수)* 기준 — 양 때문에 분류가 뒤집히지 않게(과민 회피). *열린 결정: 부족분을 랭킹에 넣을지 표시만 할지 → §14.*
- **레시피 상세·장보기**: 사용자가 고른 *인분 배수*를 곱한 필요량 기준. 장보기 담을 때 **부족분만** 담는다.

## 6. 알레르기 — allergens 컬럼 + base 상속

```
레시피 재료 R 의 알레르겐 = R.allergens ∪ (R.base_id 있으면 base.allergens)
차단 = (사용자 알레르기(표준화) ∩ 위 집합) ≠ ∅
```

- 재료 쪽은 substring 추측 없이 `allergens` 컬럼 직접 조회 — **이미 구현**(`allergyFilterV2`·`isRecipeBlockedV2`).
- **base 상속**(모짜렐라.base=치즈 → 치즈.allergens=[우유] 차단)은 **★신규/미구현** — `base_ingredient_id`와 함께 추가(상속 한 단계).
- **사용자 알레르기 입력 표준화**(땅콩↔peanut 등 동의어 매핑)도 **★신규/미구현**. 현재 `normalizeUserAllergens`는 *trim+lowercase만* 한다 — alias 매핑 상수(가칭 `ALLERGEN_CANONICAL`)는 *추후 추가*, 그 전엔 정확/대소문자 매칭만. (자동 매칭 아님 — 큐레이션된 표준화 맵.)

## 7. 왜 `forms[]`를 버리나

문서 `INGREDIENT_MATCHING_REDESIGN.md` §2-2의 `forms[]` jsonb(재료 안에 손질단계 호환행렬을 박는 모델) 대신 **가공을 `ingredient_relations` 엣지로** 표현한다.

| 기준 | `forms[]` jsonb | 엣지(relations) ← 채택 |
|---|---|---|
| 가공 표현 | 재료 안 jsonb 호환행렬 | from→to preparable_to 행 |
| 중복 | relations와 **개념 중복** (가공이 두 군데) | 단일 출처 |
| 쿼리 | "무엇이 X 될 수 있나" jsonb 순회 난해 | `WHERE to_id=X` 단순 |
| 어드민 | jsonb 편집 UI 별도 필요 | 기존 관계 승인 UI 재사용 |
| 표현력 | 편마늘→다진마늘 체인 세분화 | 엣지 여러 개로 동일 표현 가능 |
| 비용 | 재료마다 forms 입력(노가다↑) | 필요한 쌍만 엣지 |

`forms[]`가 주던 세분화(편마늘 vs 다진마늘 호환 차이)는 **실사용 가치가 낮고**(둘을 따로 보유하는 경우 희귀) 엣지로도 표현 가능 → 단순한 엣지 모델이 우월.

## 8. 구현 단계

판단 기준(§1-4): *모양 명확 → 지금*(데이터 없어도 degrade-safe), *모양 미정 → §9-B로 미룸*.
지금 만드는 인프라는 **모양이 명확한 것 전부**다. 3단계로 묶고, 각 단계는 **독립 배포**.

### 8-A. 지금 만든다 (인프라 — 모양 명확, additive·degrade-safe)

#### Phase 1 — 분류 계층 (base_id) ★ 1순위 (설계 명확 + 즉시 사용)

- [ ] **스키마**: `ingredients_master.base_ingredient_id uuid REFERENCES ingredients_master(id)` (nullable, self-FK, `ON DELETE SET NULL`).
- [ ] **무결성 가드(권장 아님·강제)**: ① `CHECK (base_ingredient_id <> id)` 자기참조 차단. ② 다단계 금지(base인 재료가 또 base를 갖지 못함) = 어드민 `set_base` 저장 시 "대상이 이미 base_id를 가지면 거부" + 순환(A→B→A) 거부 가드. 트리거보다 어드민 RPC 검증이 단순(쓰기 경로가 어드민뿐). 데이터 오염을 권장이 아닌 코드로 막는다.
- [ ] **매칭**: `matchV2`에 §5의 *2번 변형 분기* 추가 — 보유 재료의 base_id 집합도 owned 후보로. 인터페이스 불변.
- [ ] **알레르기**: `allergyFilterV2`에 base 상속 한 단계.
- [ ] **어드민**: `/admin/ingredient-matching`에 `set_base`·`add_relation` 액션 추가(기존 `link·alias_link·create_link` 옆). 한 화면에서 별칭·계층·관계 관리.

## 9. 미래 확장 여지 (지금 만들진 않되 모델이 막지 않음)

### 9-A. 모델이 자연히 수용하는 확장 (필요 시 소규모)

- **구성(made-from) 엣지**: `kind='made_from'` 추가 가능. **매칭엔 안 씀**(정직성). 용도 = ① 알레르기 전파(김치→[젓갈] 등) ② "이 재료로 뭐 만들지" 역탐색. 엣지 테이블이라 kind만 늘리면 됨.
- **재료 병합/중복정리(merge)** ★필연: 카탈로그가 크면 중복이 생긴다("대파" 따로 + "파" 따로 생성). 병합 연산 = B를 A로 흡수: `recipe_ingredients`·`user_ingredients`·`ingredient_relations`(from/to)·`shopping_list_items` 등 B 참조를 전부 A로 재지정 + B의 `aliases`를 A로 합치고 + B를 변형이면 base 재지정 + B 삭제. **어드민 필수 연산**(트랜잭션·service-role). 지금 만들진 않되 카탈로그 확장 전 도입. 데이터 무결성상 dedup 없이 키우면 매칭 분산.
- **다국어 별칭 = 큰 데이터 축**: 8개 언어 사용자가 "豚バラ"(삼겹살)·"garlic"을 쳐도 resolution 되려면 `aliases[]`에 *언어별 이름*이 들어가야 한다. 모델은 지원하나 **데이터 양이 사실상 ×8**. 정밀화 시 `{name, lang}` 구조로(현재 평문 배열). → 단계 출시(언어별 검색 고도화) 시점에 대량 데이터 작업으로 잡을 것. 지금은 한국어 동의어 위주.
- **출처(provenance) 기록**: 엣지엔 `source`(admin/author/auto)가 있으나 *컬럼*(base_id·aliases)엔 없다. "자동매칭 금지" 원칙을 감사하려면 누가·언제·자동/수동인지 필요 → 향후 `admin_audit_log`(이미 `logAdminAction` 있음) 활용 또는 `base_source`/`aliases_source` 메타. 지금은 어드민만 쓰니 audit log로 충분, 자동 유입 생기면 컬럼화.
- **가격·영양**: 변형(삼겹살)이 base(돼지고기) 영양/가격을 상속·오버라이드하는 구조로 확장 가능.
- **다단계 분류**: 삼겹살→돼지고기→육류 트리. 매칭엔 1단계면 충분하나 도감 탐색용으로 base_id 체인 순회 가능. (도입 시 §6 알레르기 상속도 1단계 → 체인 순회로 확장 필요.)
- **학습 루프**: 사용자가 대체하며 요리 → `source='auto'`·`suggestion_count` 누적 → 어드민 승급. 이미 스키마에 대비됨.

### 9-B. 진짜 deferred — *모양 미정*이라 미룬다 (데이터 부족이 이유 아님)

> 판단 기준(§1-4): 아래 둘은 "데이터가 없어서"가 아니라 **써봐야 모양이 정해지는** 것들이다. 지금 만들면 틀린 모양을 짓고 갈아엎는다(rework). ⚠️ 양·단위 비교와 대체 *비율*은 모양이 명확해 §8 Phase 2·3에서 **지금 만든다** — 아래는 그중 *모양 미정인 조각만* 떼어낸 것.

- **①-deferred: 소비 자동 추적** — "요리 완료 누르면 재료 자동 차감".
  - *왜 미루나*: 데이터가 아니라 **UX 기능**이다. 풀량 차감? 일부만? 단위 다르면 확인 모달? — 이 선택이 곧 기능인데 **사용자를 봐야 정답을 안다.** 0명일 때 추측으로 UI 짜면 실사용 시 다르게 동작해야 함 → UI 통째 재작업. (CLAUDE.md "재료 stock 추적" 로드맵 + AnyList·Whisk 1주 이탈 근거.)
  - *지금 하는 건*: 양 *비교*(있는 것끼리)는 Phase 2에서 함. 차감(소비)만 미룸.
  - *착수 trigger*: cookMode 사용자 안정화(로드맵 Phase 1) + 수요 확인.

- **②-deferred: 대체의 용도·품질 맥락** — 비율 말고 "베이킹용 vs 소스용" 같은 맥락.
  - *왜 미루나*: *비율(숫자)*은 모양이 명확해 Phase 3에서 함. 하지만 *맥락*은 **분류축 자체가 미정**이다 — {베이킹·볶음·국물}로 쪼갤지 {수분·단맛·산도}로 쪼갤지 지금은 모른다. 틀린 enum 만들면 실제 대체 사례 쌓일 때 전부 마이그레이션.
  - *지금 하는 건*: `ratio`(숫자)는 Phase 3. 자유 메모는 `notes`. 구조화된 맥락 enum만 미룸.
  - *착수 trigger*: 작성자 대체 사례가 쌓여 맥락 분류축이 데이터로 드러날 때.

## 14. 열린 결정 (구현 중 정할 것)

본문에 흩어진, *지금 단정 안 하고 구현 착수 때 정할* 항목 모음. 빠뜨림 아님 — 의도적 보류.

1. **양 부족분이 추천 *랭킹*에 영향?** vs 표시만? (§5) — 기본 제안: *표시 위주 + 약한 보조 정렬*, 1차 분류는 종류(missing 개수) 유지(양 과민 회피). 데이터 보고 조정.
2. **변형 매칭 라벨** — `owned`로 합칠지(via="삼겹살"), 신규 sub-label("변형 보유")로 구분 노출할지. UI 합의 후.
3. **가공/대체 전이** — 본 문서는 *직접 엣지만*(전이 없음)으로 확정(§5). 추후 체인 빈도 높으면 *명시적 직접 엣지 자동 생성 도구*(여전히 추측 아님)만 고려, 런타임 전이 추적은 안 함.
4. **base 1단계 vs 다단계** — 매칭은 1단계로 시작(§8). 도감 트리 수요 생기면 체인 순회(+ 알레르기 상속도 체인).
5. **변환 계수 저장 위치** — 재료별 `grams_per_ml`·`grams_per_piece`를 컬럼으로 둘지(2개 컬럼) 별도 `ingredient_units` 테이블로 둘지. 단위 종류가 늘면 테이블, 아니면 컬럼. Phase 2 착수 시 결정.
